# Лабораторная работа 2: система учета футболистов

## 1. Назначение системы
Приложение предназначено для формирования и ведения массива записей о футболистах с поддержкой:
- добавления записей через диалоговые окна;
- поиска записей по условиям варианта;
- удаления записей по условиям варианта;
- постраничного просмотра данных;
- сохранения и загрузки XML-файлов.

## 2. Предметная область и структура записи
Одна запись игрока содержит поля:
- ФИО игрока (фамилия, имя, отчество);
- дата рождения (`date`);
- футбольная команда;
- домашний город;
- состав;
- позиция.

## 3. Архитектура
Проект реализован по шаблону `Model-View-Controller`.

- `Model`: хранение данных, валидация, фильтрация, пагинация.
- `View`: главное окно, меню, панель инструментов, дочерние диалоги.
- `Controller`: обработка действий UI, связывание View и Model, файловые операции.

Основные файлы:
- `src/app/model.py`
- `src/app/view.py`
- `src/app/controller.py`
- `src/app/xml_storage.py`
- `src/main.py`

## 4. Реализованные требования
1. Главное окно и несколько дочерних диалогов реализованы.
2. Диалоги вызываются из меню, команды продублированы на toolbar.
3. Данные формируются через диалог добавления записи.
4. Поиск выполняется в отдельном диалоге, результаты выводятся в нем же.
5. Удаление выполняется в отдельном диалоге, выводится число удаленных записей или сообщение об отсутствии совпадений.
6. Текущий массив записей отображается в главном окне.
7. Сохранение выполняется в XML через DOM (`xml.dom.minidom`).
8. Загрузка выполняется из XML через SAX (`xml.sax`).
9. Используются корректные типы данных, дата хранится как `datetime.date`.
10. Реализована пагинация в главном окне и в окне поиска:
- переход на первую, предыдущую, следующую, последнюю страницы;
- смена размера страницы;
- отображение текущей страницы, общего числа страниц, числа записей на странице и общего числа записей.
11. Подготовлены демонстрационные файлы:
- `demo_data/players_set_1.xml`
- `demo_data/players_set_2.xml`

## 5. Правила валидации и поиска
1. Для ФИО допускаются только кириллические и латинские буквы обоих регистров.
2. При вводе/загрузке ФИО приводится к виду `Ххххх`.
3. В условиях поиска и удаления можно заполнять только один элемент ФИО.
4. Поиск регистронезависимый, используется нормализация регистра.

Логика поиска по варианту:
1. Группа `ФИО + дата рождения`: все заполненные условия этой группы должны совпасть.
2. Группа `позиция или состав`: логика `OR`.
3. Группа `команда или домашний город`: логика `OR`.
4. Между группами используется логика `AND`.

## 6. Хранение данных
1. Во время работы приложения записи хранятся в памяти в `PlayerRepository`.
2. После закрытия приложения данные сохраняются только при явном действии «Сохранить XML».
3. При загрузке поддерживаются кодировки `UTF-8`, `UTF-8-SIG`, `CP1251`.

## 7. Запуск проекта
Требования:
- Python 3.12+;
- внешние зависимости не требуются (используется стандартная библиотека).

Команда запуска GUI:
```bash
python -m src.main
```

## 8. Тестирование
Автотесты покрывают:
- поиск и удаление;
- валидацию ФИО и нормализацию регистра;
- корректность DOM/SAX сериализации XML;
- загрузку XML в `CP1251`.

Запуск тестов:
```bash
python -m unittest -v
```

## 9. Сценарии демонстрации системы

### Сценарий 1. Запуск и обзор интерфейса
1. Запустить приложение командой `python -m src.main`.
2. Показать главное окно, меню и toolbar.
3. Показать, что команды меню и toolbar дублируются.

Ожидаемый результат:
- открыто главное окно с таблицей и блоком пагинации;
- доступны команды добавления, поиска, удаления, загрузки и сохранения.

### Сценарий 2. Добавление записи через диалог
1. Открыть диалог «Добавить».
2. Ввести данные игрока, выбрать дату рождения через компонент даты.
3. Подтвердить добавление.

Ожидаемый результат:
- запись появляется в таблице главного окна;
- ФИО отображается в нормализованном формате `Ххххх`.

### Сценарий 3. Проверка валидации ФИО
1. В диалоге добавления ввести в поле ФИО недопустимые символы (например, цифры).
2. Подтвердить ввод.

Ожидаемый результат:
- отображается сообщение об ошибке валидации;
- запись не добавляется.

### Сценарий 4. Поиск по условиям варианта
1. Открыть диалог «Поиск».
2. Ввести одно условие ФИО и дату рождения.
3. Выполнить поиск и показать результаты.
4. Повторить поиск с заполнением «позиция или состав».
5. Повторить поиск с заполнением «команда или домашний город».

Ожидаемый результат:
- результаты отображаются в этом же диалоге;
- поиск корректно работает по правилам варианта;
- регистр символов не влияет на результаты.

### Сценарий 5. Удаление по условиям
1. Открыть диалог «Удаление».
2. Ввести условие (например, по составу или команде).
3. Выполнить удаление.

Ожидаемый результат:
- показывается сообщение о числе удаленных записей;
- при отсутствии совпадений выводится отдельное сообщение;
- таблица в главном окне обновляется.

### Сценарий 6. Пагинация
1. Заполнить или загрузить достаточное число записей.
2. Показать переходы: первая, предыдущая, следующая, последняя страница.
3. Изменить размер страницы (например, 5, 10, 20).

Ожидаемый результат:
- корректно меняются текущая страница и число страниц;
- отображаются общее число записей и число записей на текущей странице.

### Сценарий 7. Загрузка и сохранение XML
1. Загрузить `demo_data/players_set_1.xml`.
2. Проверить, что записи появились в таблице.
3. Сохранить текущий набор в новый файл через стандартный диалог.
4. Загрузить `demo_data/players_set_2.xml`.

Ожидаемый результат:
- данные корректно загружаются и сохраняются;
- формат XML читается SAX-парсером и записывается DOM-парсером.
